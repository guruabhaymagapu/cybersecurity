# Module 09 – Incident Response Lifecycle 

---

## 1. What is Incident Response?

Incident Response (IR) is the process of identifying, managing, analyzing, and recovering from security incidents such as:

- Unauthorized access
- Malware infection
- Brute-force attacks
- Data breaches
- Phishing attacks

SOC analysts follow a structured approach to handle security incidents effectively.

---

## 2. What is a Security Incident?

A security incident is any event that:

- Violates security policy
- Threatens system confidentiality
- Impacts data integrity
- Affects system availability

Examples:

- Unauthorized login
- Malware detection
- Account compromise
- Suspicious network activity

---

## 3. Incident vs Alert

| Alert | Incident |
|-------|----------|
| Notification of suspicious activity | Confirmed security event |
| Generated by SIEM tools | Verified by SOC analyst |
| May be false positive | Requires response |

SOC analysts investigate alerts to determine whether they are actual incidents.

---

## 4. Incident Response Lifecycle Phases

The Incident Response Lifecycle consists of six major phases:

1. Preparation  
2. Identification  
3. Containment  
4. Eradication  
5. Recovery  
6. Lessons Learned  

SOC teams follow these steps during incident handling.

---

## 5. Phase 1 – Preparation

Preparation involves:

- Security policies
- Incident response plan
- Log monitoring setup
- Alert rules configuration
- Backup systems
- Security awareness training

SOC analysts must be ready before an incident occurs.

---

## 6. Phase 2 – Identification

Identification involves detecting:

- Suspicious alerts
- Unauthorized activity
- Malicious behavior
- Security violations

SOC analysts:

- Monitor SIEM alerts
- Analyze logs
- Validate alerts
- Determine whether incident is true or false positive

---

## 7. Phase 3 – Containment

Containment prevents attack from spreading.

Short-Term Containment:

- Isolate affected system
- Disable compromised account
- Block malicious IP

Long-Term Containment:

- Apply patches
- Update firewall rules
- Restrict network access

---

## 8. Phase 4 – Eradication

Eradication involves removing:

- Malware
- Unauthorized accounts
- Malicious processes
- Backdoor access

SOC team ensures threat is completely removed from system.

---

## 9. Phase 5 – Recovery

Recovery involves restoring:

- Systems
- Applications
- Services
- Network access

SOC team verifies:

- Systems are functioning normally
- No malicious activity remains

---

## 10. Phase 6 – Lessons Learned

After incident resolution:

SOC team reviews:

- Root cause
- Attack method
- Detection gaps
- Response effectiveness

Security policies and detection rules may be updated.

---

## 11. Root Cause Analysis

Root cause analysis helps determine:

- How attacker entered system
- Which vulnerability was exploited
- Which system was compromised first

Helps prevent future incidents.

---

## 12. Incident Severity Levels

Incidents are categorized based on impact:

- Low
- Medium
- High
- Critical

Severity determines response priority.

---

## 13. SOC Analyst Role in Incident Response

SOC L1 Analyst Responsibilities:

- Monitor alerts
- Perform initial investigation
- Validate incident
- Collect evidence
- Escalate to L2/L3 team if required

SOC L1 does NOT perform deep remediation.

---

## 14. Incident Escalation

SOC analysts escalate incidents when:

- Incident confirmed
- Critical systems affected
- Sensitive data at risk
- Multiple systems compromised

Escalation ensures faster resolution.

---

## 15. Importance of Documentation

SOC analysts must document:

- Incident details
- Affected systems
- Attack timeline
- Investigation steps
- Actions taken

Helps in future investigation and compliance.

---

## 16. Incident Timeline Analysis

SOC analysts track:

- When attack started
- When detection occurred
- When response started
- When recovery completed

Helps understand attacker behavior.

---

## 17. Communication During Incident

SOC team communicates with:

- Incident Response Team
- System Administrators
- Management Team
- Affected Users

Ensures coordinated response.

---

## 18. Post-Incident Reporting

After recovery, report includes:

- Incident summary
- Impact assessment
- Root cause
- Response actions
- Preventive measures

---

## 19. Benefits of Incident Response Lifecycle

Helps organizations:

- Detect attacks early
- Minimize damage
- Restore systems quickly
- Prevent future attacks
- Improve security posture

---

## 20. SOC Use Case Example

Incident:

Multiple failed login attempts followed by successful login.

SOC Actions:

- Investigate alert
- Identify compromised account
- Contain affected system
- Reset password
- Escalate incident
- Document findings

---

## 21. Types of Security Incidents

Security incidents are categorized based on the nature of the threat.

Common incident categories include:

- Malware Infection
- Phishing Attack
- Brute Force / Account Compromise
- Ransomware Attack
- Insider Threat
- Data Exfiltration
- Web Application Attack
- Denial of Service (DoS)

Incident classification helps determine severity, priority, and response strategy.

---

## 22. Evidence Handling and Preservation

During incident investigation, evidence must be handled carefully.

SOC analysts must:

- Preserve relevant logs
- Avoid modifying compromised systems unnecessarily
- Record timestamps accurately
- Take screenshots when required
- Document investigation steps
- Maintain integrity of collected evidence

Proper evidence handling ensures reliable investigation and supports forensic analysis if required.

---

## 23. SLAs and Response Metrics

Security Operations Centers operate under defined response timelines.

Important concepts include:

- SLA (Service Level Agreement): Defines response time expectations.
- MTTD (Mean Time To Detect): Time taken to detect an incident.
- MTTR (Mean Time To Respond): Time taken to respond to an incident.

SOC analysts must respond to alerts based on severity level and SLA requirements.

---

## 24. Incident Documentation Format

Every confirmed incident must be properly documented.

A standard incident record includes:

- Incident ID
- Date and Time Detected
- Severity Level
- Affected System / Asset
- Incident Description
- Indicators of Compromise (IOCs)
- Investigation Steps Performed
- Actions Taken
- Escalation Details
- Final Status (Resolved / Escalated)

Proper documentation ensures accountability, audit compliance, and knowledge sharing.
